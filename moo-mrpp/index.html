MOO-MRPP

Multi-Objective Optimization for Multi-Robot Path Planning

v3.7.0 Python 3.8+ NSGA-II / NSGA-III / SPEA2 A* / RRT / RRT*

Overview

MOO-MRPP is a comprehensive research framework that implements multi-objective evolutionary optimization for multi-robot path planning. It goes far beyond a simple "A* → NSGA-II → select solution" pipeline, offering a rich architecture with configurable pathfinders, multiple optimization algorithms, diverse problem formulations, and extensive visualization capabilities.

🔧 Configurable Pathfinders

Choose from A* (deterministic, optimal), RRT (fast, diverse), or RRT* (balanced quality and diversity) for initial path generation.

🧬 Multiple Optimizers

NSGA-II for baseline, NSGA-III for many-objective problems, and SPEA2 with external archive for academic comparison.

📊 Three Formulations

Aggregate objectives only, per-agent objectives with constraints, or per-agent objectives without constraints.

🤖 Robot Models

Support for differential-drive mobile robots (DDMR 4-dir) and omnidirectional robots with proper kinematic expansion.

🚧 Moving Obstacles

Dynamic obstacles with one-time or repeat modes, time-delayed activation, and full collision integration.

📈 Interactive Dashboard

Streamlit-based web dashboard for run comparison, optimization launching, and batch experiment management.

Installation

  1. Create a Python 3.8+ virtual environment (recommended)
  2. Install dependencies:
pip install -r requirements.txt

For dashboard support, also install:

pip install streamlit plotly pillow matplotlib

For a reproducible environment, pin package versions or use a tool like pip freeze / venv / conda.

Quick Start

Single Run

python moo_multi.py run \
  --agent conf_agent4_v001.yaml \
  --config conf_proc_v007.yaml \
  --seed 1 \
  --auto-continue 1

Each run creates a timestamped folder under logs/ containing optimization artifacts (npz, csv, yaml), snapshot config files, run_history.yaml, and generated plots.

python moo_multi.py run \
  --from-log .\logs\Scenario_16-5R-Map2-agg_obj_only \
  --formulation agg_obj_only \
  --auto-continue 1 \
  --enable-console-output 0 \
  --n-max-gen 100 \
  --seed 1

Architecture

The framework operates in four conceptual layers:

  1. Configuration and Reproducibility Layer — reads config, seed, overrides, logging, and output folders
  2. Problem Construction Layer — builds environment, agents, optional moving obstacles, initial paths, and chromosome structure
  3. Evolutionary Optimization Layer — runs NSGA-II or NSGA-III with crossover and mutation operators specialized for path planning
  4. Selection and Reporting Layer — forms the Pareto front, selects one representative solution, then saves and visualizes results

Chromosome Structure

Each chromosome is a 1D integer array concatenating all agents' path encodings:

chromosome = [agent0_moves | agent1_moves | ... | agentN_moves]

Each agent segment contains:

  • Protected prefix (t_start): Leading zeros representing the agent's delayed start
  • Active segment: Sequence of movement codes:
    • 0 = stay / wait
    • 1 = right
    • 2 = left
    • 3 = up
    • 4 = down
  • Padding suffix: Trailing zeros allowing path extension during evolution

The per-robot chromosome length is fixed at ceil(1.5 × |A* path|), ensuring capacity for path modifications through crossover and mutation.

Optimization Formulations

The framework supports three configurable formulations:

1. agg_obj_only (Classic Aggregate)

F = [f1, f2, f3]
  f1: total residual distance to goals
  f2: number of inter-agent collisions  
  f3: total travel time

2. agent_obj_with_constr (Per-Agent with Constraints)

F = [travel_time_agent_1, ..., travel_time_agent_n]
G = [f1, f2]  # aggregate constraints

3. agent_obj_without_constr (Combined)

F = [f1, f2, travel_time_agent_1, ..., travel_time_agent_n]
⚠️ Important

Legacy names (hybrid_all_obj, agent_obj_no_constr) are automatically normalized to agent_obj_without_constr. N_OBJ and N_CONSTR are derived automatically from the active formulation.

Pathfinding Algorithms

Initial population sampling supports multiple pathfinders:

A* (Default)

Deterministic, optimal paths. Best for controlled experiments and reproducible baselines.

RRT

Fast, diverse, exploration-focused. Good for large open spaces.

RRT*

Balanced quality and diversity. Asymptotically optimal variant.

Configuration

# Use A* (default)
PATHFINDER: 'astar'

# Use RRT / RRT*
PATHFINDER: 'rrt'
RRT_PARAMS:
  MAX_ITERATIONS: 3000
  USE_RRT_STAR: false   # true = RRT*
  STEP_SIZE: 1
  GOAL_SAMPLE_RATE: 0.1
  GOAL_TOLERANCE: 1.5
  REWIRE_RADIUS: 3.0

The framework supports three multi-objective optimizers via pymoo:

Configuration Examples

# NSGA-II (default)
ALGORITHM:
  NAME: 'nsga2'

# NSGA-III
ALGORITHM:
  NAME: 'nsga3'
  NSGA3:
    REF_DIR_METHOD: 'das-dennis'
    N_PARTITIONS: 12
    POP_SIZE: null

# SPEA2
ALGORITHM:
  NAME: 'spea2'
💡 Note

For NSGA-III, POP_SIZE is automatically adjusted upward to be at least as large as the number of reference directions. Mismatches between N_OBJ and reference direction dimensionality fail early with a clear pymoo error.

Genetic Operators

Domain-specific crossover and mutation operators designed for path planning.

Crossover Operators

Configured in the XOVER_PARAM block. Both operators are mutually exclusive per-agent per mating.

OperatorCostDescription
Whole-Path Swap (WPS)Cheap (array copy)Swaps entire active segment between two parents
Partial-Path Swap (PPS)Expensive (A* stitching)Coordinate-domain crossover with A* gap repair

Mutation Operators

Configured in the MUT_PARAM block. Multiple mutations can fire on the same individual.

OperatorEffectDependency
Insert-Zero (INS)Path becomes longerNone
Delete-Zero (DEL)Path becomes shorterRequires INS to create zeros
Swap-Direction (SWD)Changes path shape, same lengthNone
Wait-Detour (WTD)Inserts 2-move out-and-back detourNone
Path-DetourReplaces 1 move with 3-move detourDisabled by default
⚠️ Caveat

delete_zero is structurally a no-op without insert_zero being active — enable both together or neither.

Preferred Solution Selection

An explicit decision layer chooses one representative solution from the Pareto front. The selection strategy is configured via:

PREFERRED_SOLUTION:
  STRATEGY: min_total_travel_time

Available Strategies

StrategyDescriptionBest For
min_total_travel_timeMinimizes fleet-wide sum of travel timesEfficiency-focused missions
min_max_travel_timeMinimizes maximum individual travel time (makespan)Time-critical missions
augmented_chebyshevBalanced compromise near the ideal pointMathematically balanced
knee_pointEmphasizes bend-point trade-off solutionsGeneral compromise
lexicographicPriority-based selection by objective orderStrict objective hierarchy

Safety Overrides

Before applying the requested strategy, the system filters the Pareto front to a "Safe Subset":

  1. Zero Collisions — solutions with collisions are penalized or excluded
  2. Minimum Residual Distance — solutions where agents reached their goals are prioritized

If the requested strategy fails, the system automatically falls back to Lexicographic Safe Subset selection.

Moving Obstacles

Moving obstacles are defined alongside agents in the agent config:

moving_obstacles:
  - name: movobs0
    t_start: 0
    mode: repeat          # 'one-time' or 'repeat'
    path: [[16, 5], [17, 5], [18, 5], [19, 5], [20, 5]]
  - name: movobs1
    t_start: 15
    mode: one-time
    path: [[13, 17], [13, 16], [13, 15], [13, 14]]

Key Behaviors

  • mode: one-time — travels the path once, then holds at the final position
  • mode: repeat — travels back and forth in a loop
  • t_start — timestep when the obstacle begins moving
  • Paths are internally expanded to account for rotation delays (same kinematic model as agents)
  • Visualized as dashed black overlays in path plots and dynamic markers in animations

Collision Detection

The collision checker detects:

  • Vertex collisions — two or more entities occupy the same cell at the same timestep
  • Edge collisions — two entities swap positions between consecutive timesteps

Important: only collisions involving at least one agent are counted. Obstacle-obstacle collisions are ignored.

Batch Experiment Runner

run_multi.py runs batch experiments from a YAML configuration file. A single config can contain multiple scenarios, repetition counts, seed strategies, and parameter sweeps.

Quick Start

python run_multi.py --config conf_rm/run_multi_config.yaml

YAML Format

scenarios:
  - name: "Scenario 1"
    script: "moo_multi.py"
    subcommand: "run"
    args:
      config: "conf_proc_v007.yaml"
      agent: "conf_agent5_v001.yaml"
      map: "map032_size25x25.bmp"
      formulation: "agg_obj_only"
      auto-continue: 1
      enable-console-output: 0
      n-max-gen: 2
    repeats: 3
    seed_strategy: "incremental"
    base_seed: 1

Seed Strategies

StrategyBehavior

incrementalSeed increments by 1 per repetition
randomRandom seeds for each repetition

The framework supports multiple evaluation modes:

Paper Roadmap

This framework supports an ongoing dissertation research program with six papers:

Paper #01 — Foundation Published (Conference)

Foundation approach using NSGA-II. Historical baseline for the research program.

Paper #02 — Two-Robot Baseline Published (Conference)

Baseline with two robots using NSGA-II. Proof that MOGA is viable for static MRPP.

Paper #03 — Formulation Comparison Journal

Compares three formulations: agg_obj_only, agent_obj_with_constr, and agent_obj_without_constr. Main recommendation: agent_obj_with_constr for full feasibility and interpretable per-agent trade-offs.

Paper #04 — Operator Rate Analysis Journal / JRC

OFAT and Taguchi analysis of crossover and mutation operator rates. Determines effective operator profiles.

Paper #05 — Optimizer Comparison Journal

Compares NSGA-II, NSGA-III, and SPEA2 on the selected formulation with locked operator profile.

Paper #06 — Moving Obstacles ASSET / Submitted

Validates the best method on dynamic environments with moving obstacles.

Dissertation Narrative

  1. Archive Paper #01 and #02 as published historical foundation
  2. Show formulation choice matters (Paper #03)
  3. Lock operator profile through sensitivity analysis (Paper #04)
  4. Compare optimizers with locked settings (Paper #05)
  5. Validate on dynamic environments (Paper #06)
  6. Synthesize into dissertation narrative: static simple → dynamic complex
